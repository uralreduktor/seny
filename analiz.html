1. Архитектура Решения и Технологический Стек

1.1. Общая Архитектура

Система строится на классической трехзвенной архитектуре, обеспечивая разделение ответственности, масштабируемость и гибкость в разработке. Ключевые компоненты включают слой представления на React, слой приложения на Python и слой данных на PostgreSQL с расширением pgvector.

Компоненты системы:

  • Фронтенд: React с PrimeReact для интуитивного UI
  • Бэкенд: Python (Flask/Django) для бизнес-логики
  • База данных: PostgreSQL + pgvector для векторного поиска

1.2. Технологический Стек

PostgreSQL + pgvector

Расширение pgvector превращает PostgreSQL в полноценную векторную базу данных, поддерживающую эффективный поиск ближайших соседей.

• Поддержка HNSW и IVFFlat индексов

• Косинусное и евклидово расстояние

• Интеграция с традиционными SQL-запросами

Python Backend

Flask или Django для реализации бизнес-логики, генерации эмбеддингов и выполнения сложных расчетов сходства.

• RESTful API для фронтенда

• Интеграция с ML-моделями

• Асинхронная обработка задач

React Frontend

Компонентная архитектура с PrimeReact для создания интуитивно понятного и функционального интерфейса.

• Компонент AutoComplete для поиска

• DataTable для отображения результатов

• Система управления калькуляциями

Система обрабатывает чувствительные коммерческие данные, требуя многоуровневой защиты: от фронтенд-валидации до шифрования данных при хранении и передаче.

2. Модель Данных и Методы Поиска Аналогов

2.1. Централизованное Хранилище Данных

Структура таблицы tender_items:

Поле Тип данных Описание
id UUID / BIGSERIAL Уникальный идентификатор позиции
name TEXT Краткое название позиции
description TEXT Полное техническое описание
embedding VECTOR(n) Векторное представление для семантического поиска
bom_matrix JSONB / BYTEA Матрица смежности BOM для структурного поиска
total_cost NUMERIC Итоговая рассчитанная стоимость

2.2. Семантический Поиск с Использованием pgvector

Для реализации семантического поиска используется расширение pgvector, которое позволяет PostgreSQL эффективно хранить и индексировать векторные эмбеддинги высокой размерности [33].

Генерация эмбеддингов

Python-бэкенд использует модели text-embedding от OpenAI или sentence-transformers для преобразования текстовых описаний в векторы фиксированной размерности (1536 для text-embedding-3-large).

Индексирование

Для обеспечения высокой производительности создаются HNSW-индексы, обеспечивающие быстрый приближенный поиск ближайших соседей [37].

Пример SQL-запроса для векторного поиска:

SELECT id, name, description, cost,
1 - (embedding <=> %s) AS similarity
FROM tender_items
ORDER BY embedding <=> %s
LIMIT %s;

Поддерживаемые метрики расстояния:

  • • Косинусное расстояние (<=>) - для семантического сходства
  • • L2 (евклидово) расстояние (<->) - для числовых векторов
  • • Скалярное произведение (<#>) - для альтернативных сценариев

2.3. Структурный Поиск с Использованием OPA

Для анализа структурного сходства изделий используется метод Ортогонального Прокрустеса (OPA), который позволяет количественно оценить схожесть между матрицами BOM (Bill of Materials) [117].

Процесс анализа:

  1. 1. Представление каждой позиции в виде матрицы BOM
  2. 2. Вычисление произведения матриц M = A^T B
  3. 3. Сингулярное разложение M = U Σ V^T
  4. 4. Вычисление оптимальной ортогональной матрицы R = V U^T
  5. 5. Оценка сходства d_OPA(A,B) = ||AR - B||_F

Преимущества OPA:

  • • Учитывает не только наличие компонентов, но и их количественное соотношение
  • • Обеспечивает математически обоснованную меру сходства
  • • Реализован в стандартных библиотеках (scipy.linalg.orthogonal_procrustes) [113]

2.4. Стоимостной Поиск и Фильтрация

Интеграция стоимости как критерия поиска позволяет быстро находить не просто похожие позиции, а "похожие позиции в нужном ценовом диапазоне".

Фильтрация по диапазону

WHERE price BETWEEN min_price AND max_price

Евклидово расстояние

d_cost = |price1 - price2|

3.3. Композитная Метрика Сходства

Для обеспечения наиболее релевантного поиска система использует взвешенную комбинацию трех типов сходства: семантического, структурного и стоимостного.

Формула композитной метрики:

S_total = w_sem × S_sem + w_bom × S_bom + w_cost × S_cost

где w_sem + w_bom + w_cost = 1

Преимущества подхода:

  • • Учет всех аспектов схожести позиций
  • • Гибкая настройка приоритетов через веса
  • • Более точная оценка релевантности
  • • Возможность адаптации под бизнес-логику

Многоэтапная оптимизация:

  1. 1. Быстрая фильтрация по диапазонам
  2. 2. Семантический поиск для предварительного ранжирования
  3. 3. Точные расчеты OPA и стоимостного сходства на топ-N кандидатах
  4. 4. Финальная оценка по композитной метрике
